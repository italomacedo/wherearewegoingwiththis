### Chat / NPC (this is the marquee feature — works end-to-end)
- **Cinematic chat** (`DialogSystem`): scrollable transcript seeded with prior history; `*emotes*` vs `"speech"` parsed + styled **per speaker** (player can roleplay actions or mix with dialogue); grid layout (SEND never clipped).
- **Native DOM `<input>`** (not Babylon GUI InputText) so non-US layouts / accents (ç ã é) / `?` / IME / paste work; `keydown` stopPropagation keeps typing out of the game input.
- **Pre-moderation** (`ClaudeNPCService.moderate` + `PromptBuilder.buildModerationPrompt`): a one-word ALLOW/BLOCK classifier screens the player's message BEFORE it reaches the NPC; blocked input shows a `system` line "You can't say or do that" and is never sent. Fails OPEN on CLI error. (With this gate, the earlier in-prompt violent/sexual guardrails + reply sanitizer were reverted as redundant.)
- **Windows Claude CLI launch** (`electron/main.ts` `resolveClaudeInvocation`): runs the package entry (`cli.js` via Electron-as-Node, or native `bin/claude.exe`) by reading the npm shim — no Node-on-PATH dependency. Keep Options→Claude CLI path **blank** to auto-detect.

### Health & damage (player + vehicle)
- [`src/entities/Health.ts`](src/entities/Health.ts) — pure HP value object (`applyDamage`/`heal`/`fraction`/`isDead`/`isCritical`/`toState`), shared by player and bike.
- **Player gravity + fall damage:** dismounting in mid-air drops the hero (`PlayerController.startFalling`); landing above `safeFallSpeed` costs HP scaled by impact speed. HP **0 → game over → Main Menu** (`GameWorldScene.checkGameOver`).
- **Bike crash model:** an *unpiloted* bike has its engine off, so it free-falls (no vertical drag) and crashes; impact above `safeImpactSpeed` damages it. Critical HP (≤30%) → smoke particles; **0 HP → explode** (destroyed = unmountable wreck).
- **Persisted:** `SaveGame.playerHealth` + `SaveGame.vehicle {health,destroyed}` (with `SaveService.migrate` backfilling legacy saves); carried across scenes by `GameSession`.
- HUD shows a hero HP bar (green→amber→red) + a `NAVE n%` / `NAVE DESTROYED` status (`WorldHud`).

### Controls (in game world)
- **WASD** — move, **relative to the camera** (W = where the camera faces). Hold **Shift** to sprint.
- **Z / C (hold)** — orbit the camera left / right 360° around the hero (`KEY_ORBIT_SPEED`, continuous). **Middle-mouse drag** also orbits (`ORBIT_SENSITIVITY`) but its native-canvas wiring is browser-only/untested — Z/C is the reliable, tested path. Q/E/R do not rotate.
- **E** — talk to a nearby NPC (opens chat; type `"speech"` and/or `*actions*`). **F** — enter/exit the flying bike. **Space/Ctrl** — bike altitude up/down.
- **ESC** — pause menu (Resume / Save Game / Load Game / Quit to Main Menu). While a dialog is open, ESC closes the dialog instead.
- On-screen: floating name labels over NPC/vehicle + a contextual `[E]/[F]` prompt + a persistent control hint (`WorldHud`).

New `src/core/GameSession.ts` holder (`{saveId, character, npcMemory, world, gameTimeSeconds}`) registered in ServiceLocator under `'gameSession'`. `CharacterCreatorScene.onBegin` creates+persists a save (`SaveService.createNewSave`+`save`) and registers a session; `LoadGameScene.onLoadSave` builds a session via `GameSession.fromSave`. `GameWorldScene.onEnter` adopts appearance/name/npcMemory and spawns at the saved world position (falls back to the zone spawn point when position is all-zero).
3. ✅ **Autosave (on scene exit)** — DONE. `GameWorldScene.onExit` calls `persistSession()` → `SaveService.updateWorldState` + `updateNpcMemory` (and updates the in-memory session). NOTE: this persists on *exit only*; a periodic/interval autosave during play is still future work.
4. ✅ **Real assets** — DONE (downtown V1, see Status + [ADR-0015](docs/ADR/0015-downtown-assets-and-havok-collision.md)). (The old "agent can't fetch binaries" constraint is sidestepped: Blender runs locally on user-provided source files.) ## Hard-Won Lessons (READ before debugging the running app)

These cost real debugging time — internalize them:

1. **No runtime `require()` in `src/`.** The renderer is ESM-bundled by Vite; `require` is undefined → ReferenceError → black screen. Always use static `import` at file top, even for browser-only modules (`@babylonjs/gui`, `uuid`). The `typeof document` guards keep canvas-creating calls out of Jest; merely *importing* `@babylonjs/gui` is safe in Jest.
2. **Never `await sceneManager.loadScene(next)` from inside a scene's `onEnter`.** The SceneManager is still `transitioning` from loading THAT scene, so the nested call hits the guard and silently no-ops → stuck screen. Schedule the next scene with a fire-and-forget `setTimeout` (cleared in `onExit`). See SplashScene/StudioScene/PublisherScene.
3. **Create the camera FIRST in any scene's `onEnter`,** before any slow `await` (Havok WASM, asset loads). No active camera → `Scene.render()` throws "No camera defined" every frame → black screen. Keep slow/failable init (physics) LAST and in `try/catch`.
4. **Babylon GUI alignment uses named constants, not magic numbers.** `Control.VERTICAL_ALIGNMENT_CENTER` ≠ 1 (1 is BOTTOM). Import `Control` and use the constants.
5. **`tsconfig.jest.json` is laxer than the build configs.** Jest passing ≠ typecheck passing. ALWAYS run `npm run typecheck` (checks renderer + electron + node) before committing.
6. **One Electron instance.** `vite-plugin-electron` auto-launches Electron via `onstart`; do NOT also launch it with concurrently. `npm run dev` is the single dev command.
7. **Browser-only code pattern:** guard with `if (typeof document === 'undefined') return;` + `/* istanbul ignore next */` on the browser branch, keep pure logic separate and 100% tested. This is how every system stays at coverage target without a GPU/DOM.
8. **Don't set a `layerMask` on a fullscreen GUI layer.** `AdvancedDynamicTexture.CreateFullscreenUI(...).layer.layerMask = 0x10000000` is OUTSIDE the camera's default mask `0x0FFFFFFF` → the GUI silently never renders (invisible dialog). Leave the default mask. Tests can't catch this (NullEngine renders nothing) — it only shows in Electron.
9. **Camera-relative WASD needs a +90° yaw offset.** `ArcRotateCamera.alpha` is the orbit angle of the camera *position*; the direction it *looks* is `alpha + π/2`. `CameraSystem.getYaw()` returns that offset so `PlayerController/VehicleController.computeDisplacement` point W where the camera faces. Unit tests pass with raw `alpha` because they assert with self-consistent yaw values, but the real camera default (`alpha = -π/2`) made W move sideways until the offset was added.
10. **Babylon GUI controls are drawn on the canvas, not the DOM.** DOM-based automation (Preview MCP `preview_click`, querySelector) can't click `Button`/`InputText` — they have no DOM nodes. Driving the UI for screenshots requires canvas-coordinate clicks or programmatic scene calls. Plan manual verification accordingly.
11. **Linear drag caps terminal velocity — watch it vs. damage thresholds.** With `v *= (1 - drag·dt)` the terminal fall speed is ≈ `gravity/drag` (~4.9 at drag 2). If that's below your crash-damage threshold, falls *never* deal damage. Fix: don't apply drag to the vertical axis during free-fall (engine-off bike free-falls; player fall uses no drag at all). Always sanity-check terminal speed against any speed-gated effect.
12. **NullEngine `getDeltaTime()` is ~0 in tests.** Movement/physics that integrate `dt` won't progress; assert `>=` or `jest.spyOn(engine,'getDeltaTime').mockReturnValue(100)` when a test needs real motion.
13. **Spawning the `claude` CLI from Electron on Windows: run the package entry with Electron-as-Node.** The npm `claude.cmd` shim calls `node <entry>`, but the Electron child often lacks Node on its PATH → `"...is not recognized"` (exit 1). The package entry may be `cli.js` OR a native `bin/claude.exe` (newer versions). `electron/main.ts` `resolveClaudeInvocation()` finds the shim via a synchronous PATH `whichSync('claude')`, **reads the `.cmd` shim** (`readShimEntry`) to get the true entry, and runs it: `.js` via `process.execPath`+`ELECTRON_RUN_AS_NODE=1`, `.exe` directly. Always capture+surface the child's stderr (a verbose diagnostic log `[Claude NPC] claudePath=… command=… mode=…` was decisive). Two gotchas that cost rounds: a **malformed Options path** (`…cli.jsclaude` from a bad paste) → keep Options→Claude path **blank** to auto-detect; and a **broken global install** (`bin/claude.exe` missing) fails even in a plain terminal — that's `npm i -g @anthropic-ai/claude-code`, not the game.
14. **NPC names are hidden until introduced (anti-metagaming).** `NPCAgent.getDisplayName()` returns `'Unknown'` until `revealNameIfMentioned(reply)` matches the NPC's name in its own dialogue; only then do the floating label, dialog header, and `[E] Talk` prompt show the real name. (Currently runtime-only — resets on reload; persisting the discovery flag is a follow-up.)
15. **Babylon GUI `InputText` mangles non-US keyboards.** It reconstructs text from key events, so ABNT2 `?`, accents (ç ã é), dead keys, IME and paste break. Use a **native DOM `<input>`** overlaid on the canvas (`DialogSystem.buildDomInput`), `stopPropagation` its keydown so typing doesn't drive the game, and read its `.value` on submit. Canvas-drawn GUI is fine for display, not for text entry.
16. **NPC content safety = pre-moderation gate, not prompt-only.** Screen the player's message with a one-word ALLOW/BLOCK classifier call BEFORE sending to the NPC (`ClaudeNPCService.moderate`); block out-of-policy input up front with a `system` line. Fail OPEN on CLI error so play never hard-stops. This made the per-turn in-character "deflect sexual/violent" prompt guardrails + reply sanitizer redundant — they were removed. The model's own refusal of *explicit* generation is a safety line the prompt can't (and shouldn't) override.
17. **MakeHuman/MPFB2 exports carry no fine facial shape keys, and `Apply Modifiers` strips the macro ones.** MPFB applies its detail morphs (nose/ears/lips…) to vertices, not as glTF morph targets — a clean export has **0** fine shape keys (verify with the `[Avatar] GLB morph targets (N)` log). The basemesh's helper "robe" is hidden by a Mask modifier; you must enable glTF **Apply Modifiers** to drop it, but applying a vertex-count-changing modifier **deletes all shape keys** — so you can't keep macro morphs *and* a clean mesh. Conclusion: don't build in-game facial morph sliders on a MakeHuman base. Do customization by **swapping whole GLBs** instead (one per ethnicity: `body_<gender>_<african|asian|caucasian|universal>`), tint skin via material `albedoColor`, and reserve runtime morphs for a future shape-key-bearing base. Also: register `@babylonjs/loaders/glTF` or `SceneLoader` can't read `.glb`; Babylon GUI `verticalAlignment = 2` is CENTER not bottom (Lesson 4 again). **(The "rotate `__root__` 180°" that was here is base-dependent — see Lesson 18.)**

18. **Rig + animate the hero via Mixamo, and mind the facing.** Several traps, learned end-to-end: **(a) Mixamo won't accept GLB** — only FBX/OBJ/ZIP. Export the body from Blender as FBX, auto-rig in Mixamo, re-download the rigged character **With Skin** as the new base GLB. **(b) Bone-name parity is everything** — get base AND clips from Mixamo so all bones are `mixamorig:*`; then retargeting separate clips is a pure name match (`diffSkeletonBones` reports mismatches). **(c) Mixamo names every clip `Armature|mixamo.com|Layer0`** — `loadAnimationClips` clones each `AnimationGroup` onto the base skeleton and **renames it to the manifest key** (`idle/walk/run/interact`) so `PlayerController`'s name-based playback works. Separate clips are loaded "Without Skin" (walk/run **In Place**, 30fps) and **take precedence** over any embedded in the base. **(d) Facing is base-dependent: Mixamo faces +Z, MPFB faced away.** +Z IS our world forward at `rotation.y=0`, so with `root.rotation.y = facing` you must **NOT** rotate the model — keeping the old 180° flip makes the hero "moonwalk" (body 180° from travel). The flip was removed in `assembleGltf`; the creator camera moved to `alpha=+π/2` to still face the model's front. **(e)** Convert the Mixamo FBX→GLB headless with `scripts/convert_anims.py` (`blender --background --python`). Verify with the `[Avatar] loaded anim clip "…"` / `…bones absent from the base rig` console logs.

19. **Havok WASM in Vite/Electron: copy it to `/public`, don't deep-import.** `HavokPhysics()` with no `locateFile` resolves the wrong URL → the request falls through to `index.html` → `Incorrect response MIME type` / `WebAssembly … found 3c 21 44 4f` (`<!DO`). The package's `exports` map **forbids** a deep `?url` import (`Missing "./lib/esm/HavokPhysics.wasm" specifier`), so a `scripts/copy-havok-wasm.mjs` copies it into `public/` (hooked via `predev`/`prebuild`/`preelectron:dev`; the copy is gitignored) and `PhysicsService` passes `locateFile: () => '/HavokPhysics.wasm'`. Watch for `[Physics] Havok enabled`. **Enabling physics does NOT add collision by itself** — you still need colliders + a physics-driven mover (Lesson 20).

20. **Collision = `PhysicsCharacterController` (hero) + static `PhysicsAggregate` boxes (world).** Movement was kinematic (`root.position +=`), so nothing collided even with Havok on. Fix: when `scene.isPhysicsEnabled()`, `PlayerController` drives a `PhysicsCharacterController` (capsule; `checkSupport → setVelocity → integrate(dt, support, gravity) → root.position = cc.getPosition()`, with a `capsuleHalf` feet-offset) and the kinematic path stays for headless tests. World solidity = one invisible **box** `PhysicsAggregate` (`mass:0`) per perimeter wall + per *solid* prop (derive size from `holder.getHierarchyBoundingVectors`); **roads/sidewalks/food stay walkable** (no collider) and a **floor box** gives the controller ground. Init physics **before** the zone+player so colliders/controller exist. Pure collider boxes (`CORRIDOR_COLLIDERS`) live in the catalog; all Havok code is `isPhysicsEnabled()`-guarded + `istanbul ignore`d.

21. **Placing a separate prop into a model's hole needs the hole's real offset — measure it.** Aligning MegaKit doors into building openings by eye/guess oscillated for many rounds. The fix: parse the building GLB and read the **interior-floor mesh X-centre** per model (large was off-centre at +1, others centred) — that's where the opening is. Also: the door GLB's leaf is hinged off-centre (pivot at local x=0, leaf to −1 → half-leaf `DOOR_PIVOT`), and the building's placement rotation (π on the north side) **flips local X**, so the world offset sign depends on the side. Lesson: don't eyeball repeated nudges — measure the target submesh, and account for both the prop's pivot and the parent rotation. (FBX→GLB also imports base-colour **alpha as 0** → fully transparent; `convert_assets.py` forces materials opaque. MegaKit `?url` textures are 2K and embed per-GLB → use `--maxtex 512` to avoid ~290MB of duplicated atlases.)

22. **cRPG checks: a single d100 has too much variance — use a power-ratio.** `roll×value` let a weak character beat a strong one too often. The model is now `P = atk^k / (atk^k + def^k)` (**k=2**, configurable) then **one d100; success if < P×100** (`src/systems/SkillCheck.ts`). The stat gap dominates, luck only decides close calls, and nothing is ever a hard 0/100%. `atk`=skill% if the action fits one else the governing attribute% (`CharacterStats.checkValue`); `def`=opponent value (contested, **one** roll vs P(actor)) or a fixed difficulty (unresisted, default 50). Buffs/debuffs/cover are **±N on each side's effective value** (cover +20/+40), floored so they never lock 0/100%. RNG is injected for tests. See ADR-0016.

23. **i18n: cache the locale, but `resetLocale()` in test teardown.** `I18n.t(key)` reads the locale lazily from `SettingsService.language` and **caches** it in a module var for speed. Jest isolates modules per test FILE, so the cache is fresh per file — but **within** a file any test that calls `setLocale`/`cycleLanguage` must `resetLocale()` (and `SettingsService.reset()`) in `afterEach`, or later tests see the wrong language and English assertions fail. NPC replies follow the language via `WorldSnapshot.language` (`Respond in {language}`); the moderation/action **classifiers stay English** (they emit labels, not player-facing text). See ADR-0017.

24. **Babylon GUI `InputText` is unusable for typing — use a native DOM `<input>` (Lesson 15 again).** This bit BOTH the dialog AND the character-creator name field: the name field silently took no input. The fix is the same overlay pattern (`document.createElement('input')`, fixed-position over the canvas, `stopPropagation` on keydown so typing doesn't drive the game, read `.value` on submit/BEGIN, remove the wrapper on scene exit). If you add any text-entry field, reach for the DOM input from the start. (Also: the creator preview was T-posing because it assembled the avatar but never started a clip — `start` the looping **idle** AnimationGroup after each rebuild.)

25. **A *moving* entity needs a collider that FOLLOWS it — not a separate static box you reposition.** NPCs were first given a static `PhysicsAggregate` box (mass 0) sized from the bounding box (treated like a prop, Lesson 20). But a static Havok body does NOT track its mesh, so a repositioned NPC (combat reposition, gossip walk) left the box behind — an invisible wall at the old spot, and the NPC pass-through at the new one. Rebuilding the box on each move worked but was clumsy. The clean fix is the entity's **own collision mold**: a **capsule parented to the holder** (same shape the hero's `PhysicsCharacterController` uses), an **`ANIMATED` motion-type body with `disablePreStep = false`** so Havok reads the holder transform every step and the collider follows automatically — no rebuild plumbing. Use this for anything code-moved (NPCs); reserve static boxes for things that never move (walls, parked nave).

26. **Centralize entity locomotion behind one set of primitives — don't let it fork per feature.** NPC movement had drifted into two parallel implementations (a per-frame gossip mover `stepNpcMovers` and a combat `walkAlongPath` tween), plus scattered `holder.position` / `holder.rotation` / `agent.setPosition` writes, so every new mover risked forgetting to sync the logical agent position (which drives `[E]`/proximity/camera) or the collider. Fix: one `moveNpcTo` (moves holder + logical agent together; the parented capsule follows), `faceNpc`/`faceNpcToward`, and one `startNpcWalk`/`stepNpcWalks`/`finishNpcWalk` polyline walker — shared by gossip AND combat (player included, via `combatNode`). When you find two code paths doing "the same kind of move," collapse them to a primitive before adding a third.

27. **An AnimationGroup (idle) re-evaluates its targets every frame — a one-shot transform pin won't hold against it.** A combatant kept snapping back to its modelled forward at the end of a move: setting `holder.rotation.y` once in the walk-completion callback was immediately overwritten because the looping idle clip re-asserts the avatar's authored pose each frame. Fix: keep the desired facing in a small map (`combatFacing`) and **re-assert it every frame** (`pinCombatFacings`), skipping anything mid-walk. The general rule: to hold a transform against a running animation, re-apply it on the render loop, don't set-and-forget.

28. **The NPC prompt already had a `recentEvents` channel — wire memory into existing seams instead of inventing new ones.** Making a surviving NPC "know who died" needed no prompt redesign: `PromptBuilder` already emitted "Recent events you witnessed: …", it was just always fed `[]`. Added a persisted `NPCAgent` events memory (`rememberEvent`/`getRecentEvents`, deduped+capped) populated on combat end and surfaced via `buildWorldSnapshot`. Before adding a new prompt field, check what the builder already supports.

29. **A modal that re-frames the camera must restore it on close — and weapon/label flavour rides on the EVENT, not a render-time lookup.** Two Fase-9 playtest bugs, same shape: (a) opening the corpse-loot overlay called `enterConversationMode(holder)` to frame the body, but closing it left the camera stuck — the fix is a symmetric `onClose` → `cameraSystem.exitConversationMode()` (harmless even when the overlay was opened in manage mode, where no reframing happened). Any overlay/dialog that moves the camera on open MUST undo it on close. (b) The combat log couldn't say "with Knife" because the weapon lived only on the inventory at the moment of the strike — the durable fix is to stamp the attacker's `weaponName` onto the attack **event** in `CombatEncounter`, so `CombatController`/`objectiveLogLine`/i18n carry it through without re-deriving state later. Push display facts onto the event when it happens; don't reconstruct them downstream.

30. **A flat zero-thickness plane often renders BLACK (downward/missing normals); and Babylon silently drops the 5th+ light per material.** Two Fase-10 world-lighting traps: (a) the imported `street_asphalt_9x9` road was a flat plane (ext height 0) that came in unlit/black and blew out under the flashlight — fixed by making the street a `CreateGround` (faces up → lit) or a thick slab; avoid flat decals for lit surfaces, and for road *markings* use **emissive** tiles (ignore lighting/normals entirely). (b) A held flashlight is the 5th+ light alongside the street neons, and Babylon's per-material `maxSimultaneousLights` defaults to **4**, so it was dropped on most surfaces — raise it on every material **including async-loaded ones** via `scene.onNewMaterialAddedObservable`. Also: derive a held prop's scale from its GLB's **measured bounding box** (pack meshes are authored large), and never trust a non-uniform-scale + rotation combo on an imported tile (it broke the tile's normals → black) — prefer uniform scale.

31. **Calibrate-in-engine beats guess-and-screenshot: build a tiny in-game tuning tool whose output persists.** Held-item attach offsets (pos/rot/scale/bone) are impossible to eyeball; instead of round-tripping screenshots, the **Adjust tool** (`AttachAdjuster` pure + `AdjustOverlay`, opened from an inventory "Adjust" button) lets the owner nudge the live prop with camera wheel-zoom/orbit and saves `{bone,pos,rot,scale}` per-save in `SaveGame.heldAttach` (merged over the catalog default by `resolveAttachWith`/`boneFor`). When a value can only be judged visually, give the human a knob + persistence rather than iterating blind.

32. **For a world click outside a GUI scrim, listen on the CANVAS DOM (`pointerdown`), not `scene.onPointerObservable`.** The Phase-11 out-of-combat surprise attack drew its green/red target ring fine (the preview reads `scene.pointerX/Y` every frame in `update()`), but **clicking never committed** — `scene.onPointerObservable` `POINTERTAP`/`POINTERDOWN` simply wasn't delivering the event here (the ArcRotateCamera input pipeline swallowed it). The in-combat targeting doesn't hit this because it commits through the CombatOverlay's full-screen GUI **scrim** `onPointerUpObservable`. With no scrim on foot, the reliable signal is a direct `engine.getRenderingCanvas().addEventListener('pointerdown', …)` (filter `e.button === 0`; remove it in `onExit`). Diagnosis tip: if `pointerX/Y` updates (ring follows the cursor) but no click handler fires, the input manager is alive — it's the *observable delivery* that's failing; drop to the DOM event. (Confirmed by a one-shot `console.warn` in the commit fn — clicking logged the resolved `{cand,dist,reach}` only after switching to the DOM listener.)

33. **Modular avatars: cross-mold mesh swapping works by REBINDING to one shared skeleton — and reparent BEFORE you dispose the donor's neighbours.** Quaternius "Ultimate Modular" outfits are *whole characters*, but each ships 4 separate region meshes (`{Name}_Head/_Body/_Legs/_Feet` + optional `Pistol`/`Sword`/`Backpack`) all skinned to an **identical-order** 62-bone rig *within a gender* (verified from the source glTF `skins.joints` — order, not just the set, is identical; cross-gender is NOT guaranteed → compose intra-gender only). So a modular avatar (head from A, top/Body from B, lower/Legs+Feet from C) loads up to 3 GLBs, keeps **one** donor's skeleton + the renamed loco/combat clips, and for every borrowed mesh does `mesh.skeleton = donorSkeleton` (rebind by bone index) + `setParent(donorRoot)`. Two traps that cost a playtest round: **(a)** Babylon's glTF `__root__` is **in `container.meshes`** — if you iterate-and-dispose, you can dispose the foreign `__root__` (recursive) *before* the kept Head/Bottom child is reparented out, silently disposing it too (symptom: the borrowed region is **invisible** unless its outfit happens to equal the donor/Top). Fix: reparent all kept meshes onto the donor root **first**, then dispose each foreign container **wholesale** (`rootNodes.forEach(n => n.dispose())` + its skeletons + animationGroups) — nothing is disposed before it's re-homed. **(b)** A re-skinned mesh can carry stale bounds → false frustum-cull; set `alwaysSelectAsActiveMesh = true` on borrowed meshes. Tinting goes region-aware (`tintRoleForMaterialInRegion`): semantic name wins (exposed `Skin` on a Body/Legs mesh stays skin), then a per-outfit hair override (themed molds whose mohawk is `Red`/`Red_Dark`/`Hair_Brown`, not a generic `Hair` material), then clothing → top/bottom by region. All-equal picks collapse to one load = the legacy whole-outfit look (minus the now-stripped weapon). Pure parts (`partRegionOf`/`tintRoleForMaterialInRegion`/`planModularLoad`/`resolveAvatarParts`) are 100% tested; the load/rebind/tint is browser-only `istanbul ignore`d. See ADR-0024.

34. **An UNHANDLED `stdin` stream error on the Claude CLI child crashes the whole Electron MAIN process — it looks like a random game crash with NO renderer stack.** Symptom (cost a long hunt): the game "crashed" intermittently — the user blamed the nave (flying out of bounds, descending), but every crash log was just the main-process terminal, ended right after two `[Claude NPC]` autonomy lines, and the **main-process PID itself exited** (`"process N not found"`). No `render-process-gone`/`child-process-gone` fired → it wasn't the renderer or GPU. Root cause: in `electron/main.ts` the spawn did `proc.stdin.write(prompt)` with a handler on the **child** (`proc.on('error')`) but **none on the stdin stream**; when the Claude child failed/closed early (e.g. two concurrent autonomous calls), the stdin pipe's `EPIPE` became an **uncaught exception** that killed the main process. Fixes: (a) `proc.stdin.on('error', …)` + `try/catch` the write/end; (b) a main-process safety net `process.on('uncaughtException'|'unhandledRejection')` that **logs instead of exiting** (single-player game's main process must never die from a stray async error); (c) crash-reason logging (`win.webContents.on('render-process-gone')`, `app.on('child-process-gone')`) — its *absence* in the log was the clue that pinned the blame on the main process. Lesson: when an Electron app dies with no JS stack and the main-process PID is gone, suspect an unhandled stream/'error' event in main, not the renderer — and add the process-level safety net.

35. **Audio: drive playback with plain `HTMLAudioElement`, not Babylon `Sound`; synth the engine drone with Web Audio; and relax Electron autoplay for pre-gesture music.** First audio in the project (Fase 13). Babylon's legacy `Sound` engine was never unlocked/initialized → **nothing played**. Switching SFX/music to `new Audio(url)` (one-shots + looping beds, volume = a pure bus mixer) fixed it with no scene/engine dependency; it auto-unlocks after the first user gesture. The keep-the-core-pure pattern holds: `AudioManager` mixer math + `SfxCatalog`/`MusicCatalog` (cue/track → asset, `sfxForBeat`, `musicForScene`, `fadeStep` crossfade) are 100% tested; the `new Audio`/Web-Audio playback is browser-only `istanbul ignore`d. The nave engine is a **procedural sine oscillator** (Web Audio) gliding 180→220 Hz with throttle (`setTargetAtTime`), not a sample loop. Branding/menu music must play **before any click**, so set `webPreferences.autoplayPolicy: 'no-user-gesture-required'` in `electron/main.ts` (Chromium blocks audio autoplay by default). Asset pipeline: `scripts/convert_audio.mjs` (bundled `ffmpeg-static`, no system install) normalizes freesound clips → mono OGG (SFX) / stereo OGG (music); CC-BY clips are credited in `CREDITS.md`.

36. **Anything code-moved needs its collider to FOLLOW it, the character controller fights raw `position.set`, and a yawed model needs facing compensation — the nave hit all three.** Vehicle (nave) gotchas fixed together: **(a) collider follows** — the nave's Havok box was static at spawn (invisible wall left behind when it flew/fell); parent it to the nave root with an **`ANIMATED` body + `disablePreStep=false`** so it tracks each step (same self-following mold as the NPC capsule, Lesson 25). **(b) dismount teleport** — setting `player.getRoot().position` on dismount was overridden by the `PhysicsCharacterController` (hero snapped back to spawn); use `PlayerController.teleport()` (recreates the capsule at the new spot). **(c) physics-path fall damage** — fall damage lived only in the kinematic `updateVertical`; the Havok path (`updatePhysicsMovement`) needed its own airborne→grounded impact-speed charge. **(d) moonwalk** — the model carries `VEHICLE_MODEL_YAW=π` (faces the camera parked), so steering with `root.rotation.y = facing` pointed the nose 180° off travel; compensate with `facing − VEHICLE_MODEL_YAW`. **(e) out-of-bounds = crash trigger** — flying past the walls fed extreme coords elsewhere; clamp the nave to a `horizontalHalfExtent` (the closed street). All browser-only/`istanbul ignore`d except the pure `computeFlightStep` clamp (tested).

37. **A kinematic flyer needs an INJECTED surface probe to land on geometry, and a lazy npm model dep trips two doc/test traps.** Two unrelated lessons from "land the nave on a rooftop" + Fase-13D TTS. **(a) Nave rooftop landing:** the nave integrates lift/drag/gravity and clamped to ONE global floor plane (`hoverHeight`/`groundRestHeight`), so it phased through building roofs (dismounting on a roof left the hero up there on the physics controller while the unpiloted nave free-fell through to the street and exploded). Fix: `computeFlightStep` takes a `surfaceFloor` arg (floor = surface + clearance), and `update()` reads an **injectable `floorProvider`** the scene wires to a downward `scene.pickWithRay` (origin just above the nave, excluding the nave's own meshes/`col-*` colliders + the hero; nearest solid below = floor, no-hit → 0). Pure `surfaceFloor` math + provider-driven rest tested; the raycast probe is browser-only. (Same "code-moved things need world-aware support" family as the follow-collider, but here the nave doesn't *physics-collide* — it's kinematic — so a raycast, not a body, is the right tool.) **(b) `kokoro-js` (Kokoro TTS) gotchas:** (i) a JSDoc comment containing the literal `af_*/bf_*` **closed the block comment early** (`*/`) → cascading parse errors; never write `*/` inside a `/** */` comment (use `af_/bf_`). (ii) `await import('kokoro-js')` **typechecks under the build tsconfig but FAILS under ts-jest** (its resolver can't read the package's `exports` types map → "Cannot find module") — suppress with `// @ts-ignore` (NOT `@ts-expect-error`, which errors when the build tsc has nothing to ignore). The heavy model load/synth is browser-only/`istanbul ignore`d + **fail-open**; `kokoro-js`+`@huggingface/transformers` are lazy dynamic imports so Vite code-splits them out of the main bundle (model fetched from HF on first run or vendored under gitignored `public/models/`).

38. **In an Electron renderer, a dual-build npm dep (`node` + browser exports) resolves to its NODE build → `Dynamic require of "path" is not supported`; alias it to the web build.** `@huggingface/transformers` (the Kokoro TTS dep, pulled by `kokoro-js`) ships conditional exports: a `node` condition (`transformers.node.cjs`, which `require('path')`/`fs`) and a `default` (`transformers.web.js`, onnxruntime-web/WASM). `vite-plugin-electron-renderer` makes the renderer prefer **Node** conditions (so Node APIs work), so the dev pre-bundle grabbed the node build and the first TTS call threw `Dynamic require of "path" is not supported` (esbuild's runtime stub for an unanalyzable `require`). Confusingly the **production** build picked the web build (rollup, different conditions) — so it only failed in dev. Fix: a `resolve.alias` forcing `@huggingface/transformers` → its `dist/transformers.web.js` (applies to kokoro-js's internal import too, since Vite aliases are global) + `optimizeDeps.include` both. After editing vite.config, **restart the dev server** (esbuild re-optimizes deps). General rule: when a browser/Electron-renderer dep crashes on a Node built-in, check its `package.json` `exports` for a `node` vs browser/`default` split and alias to the browser entry. (Also why the TTS load must stay **fail-open** + logged — Lesson 37/13.)

41. **Runtime avatar rebuild + Quaternius GLB gotchas (Fase 15 armor).** Three hard-won bits from making armor swap avatar regions in-game: **(a) In-game appearance changes need a real rebuild path — there wasn't one.** The creator rebuilt the avatar, but `PlayerController.spawn` assembled it once and nothing re-assembled it afterward. Added `PlayerController.rebuildAppearance(appearance)`: dispose the old `assembled`, re-`assemble`, reparent to the root (keeps root/physics-capsule/position/facing), restart the idle clip. CRITICAL follow-up: the `HeldItemRig` was built against the OLD skeleton, so after a rebuild held props attach to a dead rig — **recreate the HeldItemRig against `player.getSkeleton()` after every rebuild**, then re-sync. Keep the change render-only: armor is an **overlay** (`applyArmorOverlay` returns a CLONE with armor molds on `avatarPieces`) so the SAVED appearance is never mutated and unequipping reverts. **(b) A glTF loader can auto-play a clip on load → it loops until something else starts.** A Quaternius GLB came in with a stray clip playing (looked like a "lying down"/Death pose) that only stopped when the player pressed WASD (the first loco-state change). Fix: `assembleGltf` now `stop()`s all kept clips after load, and `PlayerController.spawn` starts the idle clip immediately (`playingState=null; updateAnimation()`), so the avatar never sits in a stray auto-played pose. **(c) Quaternius mesh names are `Cube.NNN` — region is on the NODE name, not the mesh.** To find why a specific outfit's bottom rendered invisible, read the GLB's **JSON chunk** directly (`b.readUInt32LE(12)` length at offset 12, JSON at offset 20) and list `nodes[].name`: `farmer` has `Farmer_Head/_Body/_Feet` but **no `Farmer_Legs`** → invisible as a bottom donor. Encoded as `OUTFIT_MISSING_PARTS`/`outfitProvidesPart` (creator's Bottom picker skips it; whole-outfit pick needs all 3). Don't eyeball missing geometry — parse the node list.

40. **Token economy: separate static NPC persona from the dynamic turn via `--system-prompt`; graduate to session mode early; compress classifier boilerplate.** Three levers that together reduce Claude CLI token cost by ~37–56% in a typical session: **(a) `--system-prompt` caching** — `PromptBuilder.buildStaticPersona(def, lang, cityName)` extracts the ~370-token NPC identity block (name/role/personality/backstory/routine/relationships/response rules) and passes it as a separate `systemPrompt` field through `ClaudeQueryParams` → `electron/main.ts` spawn args (`--system-prompt "..."`). `buildDynamicContext(inputs)` sends only mood/time/events/history/message as stdin. The Claude API can cache system messages across calls with the same text within 5 min → repeated turns with the same NPC are ~90% cheaper on the persona tokens. **CORRECTION (Fase 16 playtest):** the persona must ride **EVERY** turn, including `--resume` session turns. `--resume` does NOT re-apply the session-creating call's `--system-prompt`, so omitting it makes the model fall back to its default Claude Code identity (the NPC literally answers "I'm Claude Code, here to help with software engineering" mid-conversation). The text is identical each turn so the API still prompt-caches it — correctness at ~no extra cost. (`buildQueryParams` now always sets `systemPrompt`.) **(b) Early graduation** — `GRADUATION_THRESHOLD_CHARS` 6000→2500: NPCs flip to session mode (2-4 tok/turn delta) after ~2-3 exchanges instead of potentially never in short sessions. **(c) Prompt compression** — cut verbose boilerplate from classifiers (moderation, action, intent) without changing output format; saves ~200 tok/session on calls that parse ALLOW/BLOCK/VERDICT/INTENT keywords. Pattern: keep the output-format lines that the parser depends on, drop the explanatory paragraphs the model doesn't need once you trust it. Tests that check `.toContain('exact phrase')` must be updated when boilerplate changes — check for the semantic signal (e.g. 'mention being an AI'), not the full sentence. **(d) `--model haiku` + `--effort low`** on every spawned NPC call — the cheapest tier handles dialogue + trivial classifiers fine (~3-12× per-token saving), and `--effort low` (levels `low|medium|high|xhigh|max`) minimizes reasoning/thinking output tokens (cheaper + faster, ample for short NPC turns). These are the single biggest dollar levers. (Don't assume the CLI lacks an effort/reasoning knob — `claude --help` lists `--effort`; verify flags before claiming they don't exist.) **(e) The killer hidden cost: Claude Code auto-discovers the project `CLAUDE.md` (~15k tokens here) + auto-memory and injects them into EVERY `claude --print` call** when spawned from the project dir (the Electron `cwd` is inherited). Fix without breaking auth: set the child's **`cwd = os.tmpdir()`** (no CLAUDE.md to find above it). `--bare` also skips it but forces `ANTHROPIC_API_KEY` auth (no OAuth/keychain) — avoid it for OAuth/CLI-login users. Note `--system-prompt` only replaces the *default system prompt*, NOT the CLAUDE.md context — those are separate injections, so you need BOTH `--system-prompt` AND a neutral cwd. **(f) Session lifecycle: `--session-id <uuid>` CREATES a session; reusing the same id errors `Session ID ... is already in use`.** To CONTINUE an existing session use `--resume <uuid>` (`-r`). So the graduation (first session) call uses `--session-id` to create; every subsequent session turn must use `--resume` (`ClaudeQueryParams.resumeSession = !justGraduated`). This bug stayed hidden while graduation was at 6000 chars (few conversations reached session mode); dropping it to 2500 surfaced it. (Fase 14, `feat/token-economy`.)

39. **Heavy WASM/ML inference MUST run in a Web Worker or it freezes the render thread — and Vite needs `worker.format:'es'` + `import.meta` isolation for it.** Kokoro TTS synthesis (onnxruntime-web) ran on the renderer thread → ~15 s UI freeze + ~25 s latency per line. Fix: move the model load + `generate()` into a **Web Worker** (`src/workers/ttsWorker.ts`); the worker posts back the WAV as a transferable `ArrayBuffer`, the main thread only plays it (`Blob`→`HTMLAudioElement` on the voice bus). The render loop stays smooth during synthesis. Gotchas: (a) the worker dynamically imports kokoro-js/transformers (code-splitting), so Vite's default **`worker.format:'iife'` errors** ("UMD and IIFE not supported for code-splitting") — set `worker: { format: 'es' }`. (b) `new Worker(new URL('./w.ts', import.meta.url), {type:'module'})` needs `import.meta`, which **breaks the CommonJS ts-jest transform** — isolate the `new Worker(...)` call in a tiny module (`ttsWorkerClient.ts`) that's only **dynamically** imported on the browser path, so Jest never compiles it. (c) the worker + client files are never imported in tests → mark them `/* istanbul ignore file */` so `collectCoverageFrom: ['src/**']` doesn't count them as 0%. (d) try WebGPU (`device:'webgpu'`, `dtype:'fp32'`) first with a WASM/q8 fallback — much faster when the GPU path is available. Stale-utterance cancel via a token compared in the worker's `onmessage` (newer speak wins). Same "keep heavy work off the UI thread" applies to any future ML/codec/physics-heavy browser feature. **(e) kokoro-js `stream(string)` DROPS the last sentence** — given a bare string it pushes to an internal `TextSplitterStream` but never `close()`s it, so the final segment stays buffered and the async iterator hangs (N sentences → only N-1 synthesized). Fix: build a `TextSplitterStream` yourself, `push(text)` + **`close()`** (flushes the trailing sentence + ends the iterator), and pass THAT to `stream()`. General rule for streaming text splitters: always close/flush, or you silently lose the tail. **(f)** play streamed chunks via **Web Audio scheduling** (`decodeAudioData` → `bufferSource.start(cursor)` back-to-back), not one `HTMLAudioElement` per chunk — per-clip `play()` can reject (interrupted/short clip) and silently drop a sentence; schedule by a monotonic `seq` so order holds even if a later chunk decodes first, and tombstone a failed decode so the queue doesn't stall.

42. **A procedural/streamed world multiplies autonomous LLM subprocesses → the Electron MAIN process dies from spawn VOLUME, even when calls are serialized. Scope background AI to the player's CURRENT quadrant (wake/hibernate).** (Fase 17H, `feat/streaming-perf`.) Symptom: flying the nave across the mosaic crashed the **main** process (PID gone, no JS stack, no `render-process-gone`) after ~10 `claude.exe` deliberations — same fingerprint as Lesson 34 but a **different root cause**. Lesson 34's fix (swallow stdin/stream `error` + `process.on('uncaughtException')` keep-alive) was already in place; the remaining killer was simply **how many `claude.exe` get spawned**: pre-17 had 2 authored NPCs (~0.25 calls/min), but the 5×5/3×3 streamed world keeps ~20 procedural NPCs loaded, and autonomy deliberated **all of them** (each its own `ClaudeCallQueue` cooldown key → the queue dispatched them in a rapid rotation). Serializing to 1 concurrent call (an earlier attempt) did NOT fix it — it's cumulative process/handle pressure on `main`, not concurrency. **Owner-rejected** narrower fix: "only authored NPCs are autonomous." **Owner-mandated fix:** autonomy follows the player's quadrant. `NPCAgent.setAwake/isAwake` (default `true` so headless/tests stay autonomous, `NPCManager.tickAutonomy` early-returns on `!isAwake()`); `GameWorldScene.updateAwakeNpcs()` — called from `updateNpcRing()`, which `streamWorld()` invokes **only on tile change** — wakes exactly the current tile's NPC set (or the authored cast at (0,0)) and hibernates the rest. **Plus: pause autonomy entirely while piloting the nave** (`driveAutonomy` early-returns on `vehicle.isOccupied()`) — flight crosses a quadrant every few seconds, which would wake tile after tile in a burst, and you can't engage NPCs from the air anyway. Hibernating NPCs stay 100% interactive on contact (E/proximity/combat). General rule: when a world streams in unbounded agents, **bound the autonomous-LLM working set to the player's locality**, not to "everything loaded" — loading scenery is cheap, spawning subprocesses is not. **Recurrence (Fase 18):** the main process died **again** flying the nave — same no-JS-stack fingerprint, but the log's last ops were a **burst of procedural-NPC avatar builds** (the nave crosses a tile every few seconds, so `pumpNpcSpawns` streamed a full modular avatar = skeleton + skinned meshes + a Havok capsule per NPC, mid-flight). The `claude` autonomy was already paused while piloting (so that wasn't it) — it's the **native render/physics work** of building avatars during fast flight. Fix: extend the flight-pause to NPC *visual* spawning too — `updateNpcRing` skips `enqueueTileNpcs` and `pumpNpcSpawns` early-returns while `vehicle.isOccupied()`; dismount re-runs `updateNpcRing` so the current quadrant's NPCs build once on foot. You can't engage NPCs from the air anyway. General rule extends: bound BOTH the LLM AND the heavy native (mesh/skeleton/physics) per-frame work to the player's locality + activity — a fast flyer should stream cheap scenery only, deferring expensive entity construction until it stops.

43. **The save was on `localStorage` (5 MB cap) while ADR-0006 specified disk files — the procedural world's growing `npcMemory` blew the quota and the save silently vanished. Migrate to file-based IPC; keep the save lean.** (Fase 18, `feat/save-robust`.) Symptom (owner-reported): save → close → reopen, the save is **gone from Load Game**. Root cause: `SaveService` was built on `localStorage` (an interim shortcut; ADR-0006 always said JSON files via IPC), which has a **~5 MB per-origin quota**, and `save()` had **no `try/catch`** around `setItem` → once `npcMemory` grew past the cap a `QuotaExceededError` aborted the write, so nothing reached disk. Fix = implement the ADR as written: `save:list/load/write/delete` IPC handlers writing `userData/saves/*.json` (temp-then-rename = atomic; **no quota** — disk-bound). Keep `SaveService`'s **synchronous** API by **hydrating the in-memory store from disk once at boot** (`SaveService.init()`, awaited in `src/main.ts` before `game.start()`) + **write-through** on `save()`/`delete()`; `localStorage` stays as the browser-preview fallback (its `setItem` now wrapped); tests use the memory store; `init()` one-time-imports legacy localStorage saves so none are lost. **Two save-size correctness fixes rode along:** (a) `persistSession` must **merge** `this.npcMemory` (memory flushed from despawned tiles) with `serializeMemory()` (loaded agents) — it was saving only loaded agents, dropping every off-ring NPC's memory; (b) a **defeated NPC** collapses to `{defeated,inventory}` (drop conversation/disposition/ledger/events; keep corpse loot) and **persists its death** (`spawnWithMemory` re-marks it) — a killed NPC was reviving on reload, and dead NPCs' full memory bloated the save. General rule: in Electron, persist real save state to **disk via IPC**, never `localStorage` (5 MB cap + a throw aborts the write); and when the world streams unbounded entities, **bound what each save record carries** (status-only for the dead, merge what's off-screen).

44. **A crossfade that tracks only ONE outgoing element orphans music beds when fades are interrupted — the orphan loops forever and survives scene changes.** Symptom (owner-reported): combat music kept playing after the player died, the game-over bed never settled, and combat audio survived into the main menu. Root cause in `AudioManager`: `playMusic` crossfaded `old → next` but `runMusicFade` tracked a single `oldEl`; a rapid sequence (combat→gameover→menu — e.g. dying and clicking through the Game Over menu before the ~1.2 s fade finished) replaced the fade timer and **left the earlier `oldEl` mid-fade, never paused** — and it's a `loop=true` element, so it plays forever, surviving the scene transition (the fade timer lives on the AudioManager singleton, not the scene). Fix: retire EVERY displaced element into a `fadingOut[]` set BEFORE swapping; one timer fades the target up and **all** retired beds down to 0, pausing+dropping each as it hits silence; `stopMusic` pauses the current bed AND the whole `fadingOut[]` set. **General rule: a crossfade/transition manager must own *all* in-flight resources, not just the immediately-previous one — interrupting a transition is the common case, not the edge.** (Regression test uses a fake `Audio` + fake timers: combat→gameover→menu leaves only `menu` unpaused.) Architecture note: scene beds are driven by `scene:loaded`→`musicForScene`; combat/game-over are imperative overrides (`beginCombat`/`endCombat`/`checkGameOver`) on the same `musicEl` — the orphan fix is what makes every transition deterministic. (Also: a multi-combatant fight doesn't end when the player dies — `checkGameOver` must detect the dead player *combatant* and `close()` the fight, or the combat bed + turn loop run on past death.)

45. **`engine.getDeltaTime()` returns the WHOLE backgrounded gap on the first frame after Alt+Tab — clamp the per-frame delta or dt-integrated physics explodes.** Symptom (owner-reported): the parked, unoccupied nave randomly **exploded**, "seems to happen on Alt+Tab." Root cause: Chromium pauses the rAF render loop while the window is backgrounded/minimised; on focus return the first `getDeltaTime()` is the entire elapsed gap (seconds, not ~16 ms). `GameWorldScene.update()` fed that raw `dt` straight into the nave's free-fall integration — both the kinematic `computeFlightStep` and the dynamic `computeDesiredVelocity` integrate gravity by `dt`, and the Havok body's velocity is set entirely from that (gravity factor 0), so a multi-second `dt` produced a huge downward velocity → next frame it "lands" → impact ≫ `safeImpactSpeed` → crash damage → explode while parked. Fix: a pure, tested **`clampFrameDelta(ms, max=0.1)`** caps any single frame at 0.1 s (`const dt = clampFrameDelta(this.engine.getDeltaTime())`) — bounds *every* dt-consumer (nave/player fall, hunger, combat pacing, gameTime accrual, autonomy). General rule: any render-loop that integrates physics/time MUST cap the frame delta; a backgrounded tab is the canonical spike, and the cap also smooths a one-off GC/load hitch. (Pairs with Lesson 12: NullEngine `getDeltaTime()` is ~0 in tests — the opposite extreme.)

46. **A NaN passed to a Havok shape calls WASM `abort()` → the whole process dies with NO JS stack and NO crash handler; and `Math.max(floor, NaN)` is NaN, so a "floor" does NOT guard against it.** This was the ACTUAL cause of the recurring "main PID gone, no stack" nave/streaming crashes (mis-attributed to spawn volume / flight load across Lessons 34/42 and several mitigation attempts). The crash log's real tell: it died mid **NPC-capsule creation** during the avatar-build burst — `buildNpcCapsule` sized the capsule from `holder.getHierarchyBoundingVectors()`, but a **skinned mesh reports NaN/Infinity bounds for a frame mid modular-rebind** (Lesson 33's stale-bounds issue). The guard `Math.max(0.8, size.y)` looked safe but **`Math.max(0.8, NaN) === NaN`**, so `{height:NaN, radius:NaN}` reached `PhysicsAggregate(CAPSULE)` → Havok WASM `abort()` → `process.exit`, which **no `uncaughtException`/`render-process-gone`/`child-process-gone` handler can catch** (it's a native abort, not a JS throw — that's why the keep-alive nets from Lesson 34 never fired). Fix: a pure, tested **`safeCapsuleDims(sx,sy,sz)`** that returns a human default `{1.8,0.3}` on any non-finite extent and clamps to sane limits (also bake child world matrices first); same `Number.isFinite && >= min` guard added to the nave's `enableDynamicPhysics` (its `ext.x < 0.05` check also let NaN through — `NaN < 0.05` is false). **General rules:** (1) NEVER feed Havok a dimension you haven't proven `Number.isFinite` — a NaN/Inf shape is an instant unrecoverable process kill; (2) `Math.max/min` are NaN-poisoned (`Math.max(n, NaN)===NaN`), so they are NOT a NaN guard — check `Number.isFinite` explicitly; (3) when an Electron main process vanishes with no JS stack AND none of the crash handlers fire, suspect a native WASM `abort()` (Havok), not JS.

47. **An Electron app that "randomly crashes" with a CLEAN exit (code 0) is not crashing — a default Chromium keyboard accelerator is closing it. `Ctrl+W` (close window) collided with the game's `W`=forward + `Ctrl`=descend-nave bindings.** This was the real cause of the long "main PID gone, no JS stack" saga — and it cost MANY wrong fixes (NaN capsule, dt clamp, avatar-during-flight, recruit radius, BitDefender, DISABLE_PHYSICS toggle) before the right diagnosis. The tells, in order of decisiveness: (1) **empty Crashpad** despite `crashReporter.start()` → not a native crash; (2) **none** of `uncaughtException`/`render-process-gone`/`child-process-gone` fired → not a JS throw or a process abort; (3) the decisive one — adding `process.on('exit', code=>…)` + `app.on('will-quit', …)` logged **`will-quit (clean shutdown)` + `exit code=0`** → the app **quit cleanly**, it didn't die. A clean quit means a *code path* ended it; in a frameless game the only window-close path was Chromium's built-in `Ctrl+W`. The owner found it by noticing Ctrl+W closes the game (flying the nave forward-and-down presses Ctrl+W). Fix: `win.webContents.on('before-input-event', …)` → `event.preventDefault()` on `Ctrl/Cmd+W` (always) and reload (`Ctrl/Cmd+R`, `F5`) in the packaged build (kept in dev for HMR). **General rules:** (a) when an Electron app "vanishes", log the **exit code + will-quit FIRST** — a clean exit 0 is a quit, not a crash, and rules out the entire crash-hunting tree; (b) Chromium's default accelerators (`Ctrl+W`/`Ctrl+R`/`Ctrl+Shift+I`/`F5`/…) are LIVE over your canvas and WILL collide with game keybinds — block them explicitly via `before-input-event`; (c) don't ship a string of speculative "crash hardening" fixes before you've proven it's even a crash. (Several of those fixes were legit hardening and were kept, but they chased a ghost.)

### cRPG / combat model quick-reference (so a fresh context doesn't re-derive it)
- **AP** = `round(Destreza/10)` (clamped 0..`apMax`). Costs: primary (attack) 2, secondary (cover/hunker/reload) 1, **movement `moveApPerMeter` per metre** (default **0.5** → 1 AP moves 2 m). All tunable in Options.
- **To-hit** = power-ratio `SkillCheck` (k=2): `P = atk²/(atk²+def²)`, one d100 < P·100. Melee=Combate C-a-C/Força, ranged=Armas de Fogo/Destreza, defence=Percepção; cover +20/+40 on the defender.
- **Damage** = `rollWeaponDamage(stats, profile)` = `profile.damageBase + (melee:Força/10 | ranged:Destreza/20) + d(0..variance-1)`. **`FIST_PROFILE`** (base 8, var 5, range 1) == the pre-Phase-9 constants exactly; weapons come from `WEAPON_REGISTRY` via `ItemCatalog.weaponProfile(id)`.
- **Melee reach** = the weapon's `range` (fist 1 m). **Flee** only when nearest foe > `FLEE_MIN_DISTANCE` (10 m). Initiative by Destreza desc, ties by id.
- **Sides** (8B): a foe is anyone on a different `side`; `CombatRecruiter.recruitSides` assigns sides whole-scene by the NPC→NPC ledger (hostile/wary→oppose, friendly→defend, neutral/tie→out). Player flee → fight continues (`resolved`).

## Code Conventions

### Naming
- Classes: `PascalCase` — `GameManager`, `ClaudeNPCSystem`
- Files: `PascalCase.ts` for classes, `camelCase.ts` for utilities
- Interfaces/types: `PascalCase` with `I` prefix only for interface contracts (`IScene`), not for data shapes
- Private fields: no underscore prefix — use `private` keyword
- Constants: `SCREAMING_SNAKE_CASE` for module-level constants

### Patterns
- **Singleton:** Use `static getInstance()` + `static resetInstance()` (for test isolation) — see `GameManager`
- **Service registration:** Register in `GameManager.initialize()`, retrieve via `ServiceLocator.get<T>(key)`
- **Events:** Typed via `GameEvents` interface in `EventBus.ts` — add new events there first
- **Async scenes:** `onEnter()` and `onExit()` are async — await them fully before continuing
- **No God objects:** Each system has one responsibility. Cross-system communication goes through EventBus.

### Testing
- Always use `NullEngine` for Babylon.js tests — never require a real GPU
- Always call `ServiceLocator.clear()` in `afterEach`
- Always call `engine.dispose()` and `scene.dispose()` in `afterEach`
- Mock `child_process.spawn` for Claude CLI tests — never spawn real processes in tests
- Coverage must stay ≥95% lines/functions, ≥90% branches on every PR

## How to Create a New Scene

1. Create `src/scenes/MyScene.ts` extending `BaseScene`
2. Implement `onEnter()`, `onExit()`, optionally `update()`
3. Add the scene name to `SceneName` union type in `SceneManager.ts`
4. Register the factory in `src/main.ts` or the scene that transitions to it
5. Write unit tests in `tests/unit/scenes/MyScene.test.ts` using `NullEngine`

```typescript
export class MyScene extends BaseScene {
  async onEnter(): Promise<void> {
    // build scene content here
  }
  async onExit(): Promise<void> {
    // cleanup animations/timers
  }
}
```

---

## How to Create a New System

1. Create `src/systems/MySystem.ts`
2. Register it in `ServiceLocator` during `GameManager.initialize()`
3. Systems communicate only through `EventBus` — never import other systems directly
4. Write tests in `tests/unit/systems/MySystem.test.ts`

## How to Add an Asset

1. Search using Sketchfab MCP or Poly Haven API (see `docs/ADR/0005-asset-pipeline.md`)
2. Present 3 options to the user before downloading
3. Place downloaded file in `src/assets/[category]/[name].glb`
4. Create a typed reference in `src/assets/AssetManifest.ts`
5. Document the choice in an ADR update
